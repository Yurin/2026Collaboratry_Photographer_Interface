# 一人人物の参照写真再現撮影に向けたユーザー実験追加要件

## 追加要件の全体方針

本追加開発では、既存の二端末型撮影システムを「一人の人物が一枚の参照写真を再現する」という比較可能な課題に限定し、撮影者には構図、被写体にはポーズという、それぞれが直接修正できる情報を提示する。

研究上の中心は新しい認識モデルの開発ではなく、認識結果を二者へどう分配し、協調行動をどう支えるかというインターフェース設計に置く。既存モデルとルールベース判定を利用し、条件A「支援なし」、条件B「撮影者のみ支援」、条件C「役割別の双方支援」を同一基盤で切り替え、撮影結果、行動ログ、所要時間、主観評価を比較できる状態を目標とする。

本書は既存の [REQUIREMENTS.md](REQUIREMENTS.md) に対する追加要件である。既存IDとの重複を避けるため、`FR-R05`、`FR-P10`、`FR-S08`、`FR-L08` から続番を使用する。

### 現在の実装到達点

Phase 1の基盤として、実験セッション作成、条件A/B/C、試行作成・開始・完了・中断、役割別の表示制御、Web・iOSの操作ログ、通信失敗時の一時保存、試行データのJSON/JSONL保存を実装済みである。自動的な構図差分・ポーズ差分の算出と指示生成はPhase 3・4の対象である。

## 1. 追加開発の目的

### 1.1 一人人物の参照写真再現に限定する理由

複数人撮影では、各人物の姿勢、人物間距離、重なり、視線、撮影者との会話などが同時に変化し、支援効果の原因を切り分けにくい。一人人物に限定すると、人物中心、人物サイズ、身体各部の位置といった評価対象を明確にでき、参照写真と撮影結果の差分を定量化しやすい。

また、一人の被写体と一人の撮影者という最小の協調単位に絞ることで、「誰がどの差分を直すべきか」という役割分担そのものを検証できる。これは、将来複数人へ拡張する前に、二者協調型支援の基礎的な有効性を確認するために必要である。

### 1.2 撮影者と被写体に異なる支援を提示する理由

参照写真との差分には、撮影者がカメラ操作で修正できるものと、被写体が身体操作で修正できるものがある。

- 撮影者が修正しやすい要素:
  - 人物の画面内位置
  - 人物の写る大きさ
  - カメラの上下左右
  - カメラの傾き
  - 被写体との距離
- 被写体が修正しやすい要素:
  - 頭や顔の位置
  - 肩、腕、手の位置
  - 腰、重心、脚の向き
  - 身体全体の向き

両者へ同じ情報を提示すると、どちらが修正すべきか分からない、同時に動いて誤差が増える、相手の担当領域まで気にする、といった協調上の混乱が起こり得る。そのため、差分を操作主体ごとに分配し、それぞれに必要最小限の支援を出す。

### 1.3 1台完結型支援との差分

既存の1台完結型カメラコーチは、主に撮影者のカメラ画面へ構図やポーズ情報を集約する。その場合、被写体は撮影者から口頭で指示を受ける必要があり、被写体自身が差分を直接確認できない。

本システムでは、撮影者端末と被写体端末を接続し、同じ認識状態から役割別の表示を生成する。撮影者はカメラ操作に集中し、被写体は自分の身体調整に集中できる。この「共有状態を基にした支援情報の分配」と「二者が並行して修正できること」を、1台完結型との差分およびHCI上の新規性として位置付ける。

### 1.4 ユーザー実験で検証する事項

主要な研究課題は次のとおりとする。

- RQ1: 撮影者向け構図支援は、支援なしと比較して参照写真との構図差を小さくするか。
- RQ2: 撮影者と被写体への役割別支援は、撮影者のみへの支援と比較してポーズ差を小さくするか。
- RQ3: 役割別支援は、撮影時間、撮り直し回数、発話・指示負担を減らすか。
- RQ4: 役割別支援は、操作主体の明確さ、協調感、納得感、使いやすさを高めるか。
- RQ5: 支援の推定誤差や表示遅延が、利用者の信頼と撮影行動へどのような影響を与えるか。

### 1.5 主な評価指標

| 区分 | 指標例 |
|---|---|
| 撮影結果 | 人物中心誤差、人物サイズ誤差、傾き誤差、主要関節点誤差、シルエット重なり、共有された撮影写真群 |
| 作業効率 | 試行時間、撮影枚数、撮り直し回数、支援提示回数、収束までの時間 |
| 協調行動 | Ready遷移、両者の修正タイミング、指示対象、支援への追従、必要に応じた発話回数 |
| 主観評価 | 使いやすさ、役割明確性、負担、信頼、協調感、結果への満足度 |

## 2. ユーザー実験条件

### 2.1 条件設計の共通原則

- 条件間で参照写真、撮影時間上限、撮影・送信機能、画面レイアウトの基本構造を可能な限り揃える。
- 操作する独立変数は、主として「誰にどの支援を表示するか」とする。
- 条件を示す名称や説明から、参加者が優劣を推測できないようにする。
- 条件順序は参加者ペア間でカウンターバランスする。
- 実験者だけが条件IDを確認でき、参加者画面には原則として `A/B/C` を直接表示しない。
- すべての条件で同じログ基盤を動かし、非表示条件でも認識精度評価に必要な推定値は、同意範囲内で記録できるようにする。

### 2.2 条件A: ガイドなしの通常撮影

| 項目 | 内容 |
|---|---|
| 目的 | 通常の二者撮影を基準条件として測定する |
| iOS表示 | 参照写真、接続・Ready状態、撮影結果。ライブ映像および自動ポーズ指示は表示しないことを基本とする |
| Web表示 | カメラ映像、撮影、撮影済み写真、送信。構図ガイドと自動指示は表示しない |
| 使用可能機能 | QR参加、撮影、撮り直し、送信、結果確認 |
| 使用不可機能 | 自動構図指示、自動ポーズ指示、ガイドオーバーレイ、遠隔ガイド調整 |
| 主なログ | 開始・終了、撮影時刻、撮影枚数、撮り直し、Ready、共有写真、推定結果、エラー、アンケート |

条件Aでも参照写真を双方が事前に確認できるようにする。ただし撮影中に参照写真を常時表示するかは、実験計画で統一し、条件間で同じにする。

### 2.3 条件B: 撮影者だけに支援を表示する撮影

| 項目 | 内容 |
|---|---|
| 目的 | 撮影者向け構図支援単独の効果を測定する |
| iOS表示 | 条件Aと同等の基本画面。被写体向け自動ポーズ指示は表示しない |
| Web表示 | カメラ映像、目標人物枠または構図ガイド、方向指示、人物サイズ指示、撮影機能 |
| 使用可能機能 | 条件Aの機能、撮影者向け構図ガイド、構図指示、ガイドON/OFF |
| 使用不可機能 | 被写体向け自動ポーズ指示 |
| 主なログ | 条件Aのログ、構図誤差、構図指示内容・提示時刻、ガイド表示操作、指示後の誤差変化 |

条件Bでは、撮影者が必要に応じて被写体へ口頭指示を出すことを禁止しない。発話を評価対象にする場合は、実験者の観察記録または別途同意を得た音声記録を利用する。

### 2.4 条件C: 撮影者と被写体に役割別支援を表示する撮影

| 項目 | 内容 |
|---|---|
| 目的 | 二端末へ役割別に支援を分配する効果を測定する |
| iOS表示 | ライブ状態、被写体向けポーズ指示、対象部位、目標方向、Ready状態 |
| Web表示 | 条件Bと同じ撮影者向け構図支援。被写体向け詳細指示は表示しない |
| 使用可能機能 | 条件Bの機能、被写体向けポーズ指示、双方のReady共有、役割別状態同期 |
| 使用不可機能 | 相手の担当情報を常時詳細表示する機能 |
| 主なログ | 条件Bのログ、ポーズ誤差、部位別指示、被写体のReady、構図・ポーズの同時収束状態 |

### 2.5 条件別表示マトリクス

| 機能・表示 | 条件A | 条件B | 条件C |
|---|:---:|:---:|:---:|
| 撮影・送信 | ○ | ○ | ○ |
| 参照写真の事前確認 | ○ | ○ | ○ |
| 撮影者向け目標枠 | － | ○ | ○ |
| 撮影者向け構図指示 | － | ○ | ○ |
| 被写体向けポーズ指示 | － | － | ○ |
| ガイドON/OFF | － | ○ | ○ |
| Ready共有 | ○ | ○ | ○ |
| 認識・差分ログ | ○ | ○ | ○ |
| 指示提示ログ | 該当なし | 撮影者のみ | 撮影者・被写体 |

## 3. 追加機能要件一覧

優先度は、`P0` をユーザー実験成立に必須、`P1` をパイロット実験の品質向上に必要、`P2` を本実験または発展評価向けとする。

### 3.1 FR-E: 実験モード要件

| 要件ID | 要件名 | 優先度 | 対象画面 | 内容 | 理由 | 受入条件 |
|---|---|:---:|---|---|---|---|
| FR-E01 | 実験モード起動 | P0 | iOS / Backend | 通常利用と実験利用を分離し、実験モードでは参加者・条件・試行情報を必須とする | 通常機能の状態が実験データへ混入するのを防ぐため | 実験モードで必要情報がない場合は試行を開始できず、通常モードは従来どおり利用できる |
| FR-E02 | 実験条件設定 | P0 | iOS / Backend | 実験者が条件A/B/Cを設定し、セッションへ固定する | 条件の誤設定を防ぎ、再現可能にするため | 試行開始後は実験者操作なしに条件が変わらず、両端末が同じ条件を受信する |
| FR-E03 | 条件別機能制御 | P0 | 共通 | `conditionId` に基づき、表示、ガイド、指示生成、操作可能機能を切り替える | CSS上の非表示だけでは条件違反や意図しない情報提示が起こり得るため | Aでは支援なし、BではWebのみ、CではWebとiOSに支援が表示されることを実機で確認できる |
| FR-E04 | 試行ライフサイクル | P0 | 共通 | 試行を `configured`、`ready`、`running`、`captured`、`completed`、`aborted` で管理する | ログの開始・終了点と撮影結果を一意に定めるため | 不正な状態遷移が拒否され、全遷移が時刻付きで保存される |
| FR-E05 | カウンターバランス支援 | P1 | iOS / Backend | ペアごとの条件順序と参照写真順序を登録・表示する | 順序効果と学習効果を抑えるため | 実験者が次の条件・参照写真を確認でき、実施順がログに残る |
| FR-E06 | 練習試行 | P1 | 共通 | 分析対象外として識別される練習試行を作成できる | 参加者が操作に慣れる前のデータ混入を防ぐため | `isPractice=true` の試行を実施でき、分析用出力で除外できる |
| FR-E07 | 試行中断・再試行 | P0 | iOS / Backend | 実験者が理由を付けて試行を中断し、新しい `trialId` で再試行できる | 接続不良や認識失敗時に実験全体を失敗させないため | 中断試行が上書きされず `aborted` と理由が残り、再試行が別IDで開始される |
| FR-E08 | 実験者用確認表示 | P1 | iOS | 参加者には見せない領域で、ID、条件、ログ送信状態、接続状態を確認できる | 条件取り違えとログ欠損をその場で発見するため | 実験者操作でのみ開け、撮影中の参加者画面には表示されない |

### 3.2 FR-R: 参照写真・目標姿勢要件

| 要件ID | 要件名 | 優先度 | 対象画面 | 内容 | 理由 | 受入条件 |
|---|---|:---:|---|---|---|---|
| FR-R05 | 参照写真ID管理 | P0 | Backend | 参照写真へ不変の `referenceImageId`、バージョン、縦横比、利用条件を付与する | 条件間で同一刺激を使用したことを検証するため | 各試行と共有写真群から使用した参照写真を追跡できる |
| FR-R06 | 一人人物検証 | P0 | Python / iOS | 参照写真から検出された人物数を確認し、0人または複数人の場合は警告する | 今回の分析前提を保証するため | 1人検出時のみ自動確定し、それ以外は画像変更または実験者承認を要求する |
| FR-R07 | 目標特徴抽出 | P0 | Python | 参照写真から人物枠、人物中心、人物サイズ、シルエット、関節点、信頼度を抽出する | 構図差とポーズ差を共通表現で計算するため | 抽出結果が画像座標ではなく正規化座標と信頼度を含むJSONで保存される |
| FR-R08 | 参照座標の正規化 | P0 | Python / Backend | 参照特徴を3:4基準の0〜1座標へ変換し、左右反転状態も記録する | 端末解像度やクロップ差に依存しない比較を行うため | 異なる解像度で同じ参照画像を処理しても許容誤差内で同じ正規化値になる |
| FR-R09 | 関節点手動補正 | P0 | iOS | 頭、肩、腰等の関節点を参照写真上で移動・追加・無効化できる | 既存モデルの誤推定を実験刺激へ持ち込まないため | 補正前後を表示でき、保存・再読込後も補正位置が維持される |
| FR-R10 | 目標姿勢バージョン保存 | P0 | Backend | 自動推定値、手動補正値、補正者、補正時刻、採用版を保存する | 刺激作成の再現性と分析の透明性を確保するため | 元の推定結果を失わず、各試行が使用した版を追跡できる |
| FR-R11 | 左右対応設定 | P1 | iOS / Python | 参照写真とライブ映像のミラー表示を考慮し、左右関節の対応を確定できる | 左右逆の誤指示を防ぐため | テストポーズで右手・左手の指示が表示方式にかかわらず正しい対象へ出る |
| FR-R12 | 参照写真事前検証 | P1 | iOS / Backend | 実験前に検出品質、全身の可視性、遮蔽、必要Levelへの適合を確認する | 実験中の推定不能を刺激選定段階で減らすため | 不適合理由を表示し、利用可能な支援Levelが参照写真ごとに記録される |

### 3.3 FR-C: 撮影者向け構図支援要件

| 要件ID | 要件名 | 優先度 | 対象画面 | 内容 | 理由 | 受入条件 |
|---|---|:---:|---|---|---|---|
| FR-C01 | ライブ人物枠推定 | P0 | Python / Backend | 撮影中フレームから一人の人物枠、中心、幅、高さ、信頼度を取得する | 構図差判定の最小入力となるため | 対応端末で処理結果を時系列取得でき、人物未検出も明示される |
| FR-C02 | 構図差分計算 | P0 | Backend / Python | 参照とライブの人物中心、人物サイズ、傾きを正規化座標で比較する | 撮影者が修正できる要素を分離するため | 既知のずれを与えたテスト画像に対し、方向と大小関係が正しく算出される |
| FR-C03 | 方向指示生成 | P0 | Backend | 水平・垂直誤差を閾値と比較し、「カメラを右へ」等の定型指示へ変換する | 解釈の一貫した支援を低コストで生成するため | 閾値外で正しい方向、閾値内で「位置OK」が返る |
| FR-C04 | 距離・サイズ指示生成 | P0 | Backend | 人物サイズ差から「近づく」「離れる」を生成する | 参照写真の人物占有率を再現するため | 参照より人物が小さい場合は近づく、大きい場合は離れる指示になる |
| FR-C05 | 傾き補正指示 | P1 | Web / Backend | 信頼できる水平基準または肩線等から傾きを推定し、回転方向を示す | 中心・サイズ以外の構図差を扱うため | 規定角度以上で方向付き指示、範囲内で非表示またはOKになる |
| FR-C06 | 目標枠オーバーレイ | P0 | Web | 参照人物枠をライブ映像上へ重ね、現在人物枠との関係を示す | 文章だけでなく直接操作可能な視覚手掛かりを与えるため | 条件B/Cのみ表示され、3:4クロップと座標が一致する |
| FR-C07 | 指示選択と安定化 | P0 | Backend | 同時に多数の指示を出さず、最大1件の主指示を優先度・誤差量・継続時間で選ぶ | 指示の点滅と認知負荷を抑えるため | 認識値が閾値付近で揺れても、設定時間内に指示が過剰反転しない |
| FR-C08 | 構図Ready判定 | P0 | Backend / Web | 中心、サイズ、必要に応じ傾きが許容範囲で一定時間維持されたらReadyとする | 撮影可能状態を二者で共有するため | 設定した保持時間後にReadyとなり、範囲外で解除される |
| FR-C09 | 認識不能時表示 | P0 | Web | 人物未検出または信頼度不足時は誤指示を出さず、検出不能状態を表示する | 誤った支援への追従を防ぐため | 低信頼度フレームでは方向指示が停止し、利用者向け復帰案内が出る |

### 3.4 FR-P: 被写体向けポーズ支援要件

| 要件ID | 要件名 | 優先度 | 対象画面 | 内容 | 理由 | 受入条件 |
|---|---|:---:|---|---|---|---|
| FR-P10 | ライブ姿勢推定 | P0 | Python / Backend | 撮影中フレームからシルエット、主要関節点、各信頼度を取得する | 被写体向けポーズ差判定の入力とするため | 対応Levelに必要な値を時系列取得でき、欠損関節を区別できる |

## 4. 役割別支援内容

### 4.1 支援情報の分配原則

1. 撮影者にはカメラを動かすことで修正できる情報を提示する。
2. 被写体には自分の身体を動かすことで修正できる情報を提示する。
3. 一度に提示する主指示は各画面1件を基本とする。
4. 誤差が小さい状態では新しい指示を出さず、Readyまたは「そのまま」を示す。
5. 低信頼度時は推測で指示を出さず、検出不能または上位Levelへのフォールバックを示す。
6. 指示の更新には閾値、ヒステリシス、最小表示時間を設け、点滅や反転を防ぐ。
7. 指示文はルールベースの定型文を正式な分析対象とする。

### 4.2 撮影者向け構図支援

| 差分 | 判定例 | 主な表示 | 操作者 |
|---|---|---|---|
| 人物が目標より左 | 現在中心Xが目標中心Xより小さい | 「カメラを左へ」または方向矢印 | 撮影者 |
| 人物が目標より右 | 現在中心Xが目標中心Xより大きい | 「カメラを右へ」 | 撮影者 |
| 人物が目標より上 | 現在中心Yが目標中心Yより小さい | 「カメラを上へ」 | 撮影者 |
| 人物が目標より下 | 現在中心Yが目標中心Yより大きい | 「カメラを下へ」 | 撮影者 |
| 人物が小さい | 人物枠面積または高さが目標未満 | 「近づく」 | 撮影者 |
| 人物が大きい | 人物枠面積または高さが目標超過 | 「離れる」 | 撮影者 |
| 画面が傾いている | 基準線の角度が許容値外 | 「右回りに傾きを直す」等 | 撮影者 |
| 枠が一致 | 中心・サイズ・傾きが許容範囲 | 「構図OK」 | 撮影者 |

カメラ移動方向の文言は、人物をどちらへ動かすかではなく、撮影者が端末をどちらへ動かすかに統一する。実装前に実機で方向対応を確認し、前面カメラやミラー表示による逆転を避ける。

### 4.3 被写体向けポーズ支援

| 対象 | 差分例 | 指示例 |
|---|---|---|
| 顔・頭 | 頭中心が目標より低い | 「顔を少し上へ」 |
| 肩 | 左肩が目標より低い | 「左肩を少し上げる」 |
| 腕 | 肘角度が目標より閉じている | 「左肘を少し開く」 |
| 手 | 手首が目標より内側 | 「右手を少し外へ」 |
| 腰・重心 | 腰中心が目標より右、肩腰軸が不一致 | 「腰を少し左へ」 |
| 脚 | 膝・足首方向が目標と異なる | 「左足を少し外へ」 |
| 身体の向き | 肩幅・左右関係が目標と異なる | 「身体を少し右へ向ける」 |
| 許容範囲内 | 対応Levelの主要誤差が小さい | 「ポーズOK。そのまま」 |

左右表現は原則として被写体自身の左右を用い、画面内にも「あなたの右手」など明示する。参照写真が鏡像の場合は事前設定に基づいて対応を変換する。

### 4.4 段階的な支援Level

#### Level 1: 人物中心・人物サイズ・シルエット

- 使用特徴:
  - 人物枠
  - 人物中心
  - 人物占有率
  - 粗いシルエット
- 撮影者支援:
  - 上下左右
  - 近づく・離れる
  - 目標枠
- 被写体支援:
  - 身体全体を左右へ
  - 粗い身体の向き
  - シルエットの大きな不一致
- 位置付け:
  - P0の最小実験可能構成
  - 詳細関節が不安定でも動作する

#### Level 2: 頭・肩・腰などの主要点

- 使用特徴:
  - 頭中心
  - 左右肩
  - 左右腰
  - 肩線・腰線
- 被写体支援:
  - 顔の上下左右
  - 肩の高さ
  - 腰・重心
  - 身体の向き
- 位置付け:
  - 条件Cの役割別支援を明確に評価する推奨構成

#### Level 3: 肘・手首・膝・足首

- 使用特徴:
  - Level 2に加え、肘、手首、膝、足首
- 被写体支援:
  - 腕の角度
  - 手の位置
  - 膝の曲げ
  - 脚・足の方向
- 位置付け:
  - 推定品質と処理速度がパイロット試験で十分な場合に追加する
  - Level 3が不安定でも、Level 1または2で本実験を実施可能とする

### 4.5 既存モデルの利用方針

- 独自AIモデルの学習は行わない。
- 参照写真とライブ映像へ、同系統または比較可能な既存モデルを適用する。
- 人物枠には既存の人物検出モデル、シルエットには既存のセグメンテーションモデル、関節点には既存の姿勢推定モデルを用いる。
- モデル名、バージョン、重み、信頼度閾値、入力解像度を実験設定として記録する。
- 参照写真側は実験前に人手で検証し、必要に応じて関節点を補正する。
- ライブ側の低信頼度結果は無理に補完せず、詳細指示を停止またはLevelを下げる。
- 指示判定は、正規化座標差、角度差、面積比、信頼度、継続時間を使う決定的なルールとする。
- LLMを使用する場合、決定済みの対象部位・方向・強度を自然な文へ言い換える用途に限定する。
- LLMの出力によって指示対象、方向、実験条件、Ready判定を変更してはならない。
- 条件比較の主分析では、定型文版を使用する。LLM言い換えは別実験または探索的評価として扱う。

## 5. UI要件

### 5.1 被写体側iOS画面

#### 実験条件の選択画面

- 実験者モードからのみ開けること。
- `participantId`、`pairId`、条件、試行番号、練習・本番、参照写真を確認できること。
- 条件名は参加者向け画面に直接表示せず、「撮影1」等の中立表現へ置換できること。
- 開始前に必須ID、両端末接続、参照姿勢、ログ保存先を検証すること。
- 条件変更は試行開始前だけ許可すること。

#### 参照写真確認画面

- 3:4の最終クロップを表示すること。
- 参照写真の向き、全身・上半身、必要Levelへの適合を示すこと。
- 撮影者と被写体が同じ参照写真を確認済みかをReadyで示すこと。
- 実験プロトコルに応じ、撮影開始後の参照写真表示可否を条件間で統一すること。

#### 目標ポーズ確認・補正画面

- 参照写真上に人物枠、シルエット、関節点をレイヤー表示すること。
- レイヤーごとにON/OFFと透明度変更ができること。
- 関節点をドラッグし、誤検出点を無効化できること。
- 自動推定値と補正値を区別して表示すること。
- 被写体自身の左右と画像上の左右を明記すること。
- 保存前に必要関節点の不足と低信頼度を警告すること。

#### 撮影中ライブ確認画面

- 条件Cでは、被写体向けポーズ支援とReady状態を最優先表示すること。
- 条件A/Bでは、自動ポーズ指示を表示しないこと。
- ライブ画像を表示する場合、撮影者向け構図指示を被写体へ詳細表示しないこと。
- 接続断、人物未検出、低信頼度、再接続中を参加者向けの短い文で示すこと。
- 開発者向けURL、JSON、信頼度の生値、スタックトレースを表示しないこと。

#### 被写体向けポーズ指示表示

- 対象部位、方向、必要に応じ強度を一つの主指示として表示すること。
- テキストに加え、身体アイコン、部位ハイライト、矢印のいずれかを併用すること。
- 指示が変わったことを色だけに依存して伝えないこと。
- 「ポーズOK」と検出不能を明確に区別すること。
- 最小表示時間を設け、短時間に読めない切り替わりを防ぐこと。

#### 撮影者側の準備状態表示

- 初期のユーザー実験では、自動構図Ready・自動ポーズReadyを必須にせず、進行管理に必要な簡易Readyを優先すること。
- 簡易Readyでは、撮影者接続済み、カメラ起動済み、ライブ共有中、撮影済み写真あり、写真送信済み、被写体側写真受信済みを区別すること。
- 相手待ちの場合、誰の何を待っているかを示すこと。
- 条件Cの自動Readyを導入する段階では、自分のポーズReadyと撮影者の構図Readyを別々に表示すること。

#### 撮影結果確認画面

- セッション内の撮影写真だけを表示すること。
- 撮影順、撮影時刻、送信状態を表示すること。
- 現時点では最終写真を1枚選択せず、撮影者が送信した全写真を被写体へ共有すること。
- `finalPhotoId` は将来、最終成果物を1枚に絞る実験設計へ変更する場合の任意項目とし、空でも試行完了を妨げないこと。
- 再試行する場合、元試行を上書きせず新しい `trialId` を作ること。

#### 実験後アンケート画面

- 条件終了直後に、その条件の回答画面を表示すること。
- 役割に応じた設問を表示すること。
- 回答尺度の左右方向と文言を統一すること。
- 未回答を許可する場合は欠測として明示し、0点として保存しないこと。
- 回答送信完了とローカル一時保存状態を表示すること。

### 5.2 撮影者側Web画面

#### 実験セッション参加画面

- QR参加後、ペア・試行に対応する中立的な案内を表示すること。
- カメラ権限、ネットワーク、セッション状態を開始前に確認すること。
- 撮影者役割として登録できた後に撮影画面へ進むこと。
- 参照写真の確認が必要な場合、撮影開始前にのみ統一して表示すること。

#### カメラ映像画面

- 3:4の分析対象範囲を明示し、撮影画像と表示クロップを一致させること。
- 条件Aではガイド・自動指示を表示しないこと。
- 条件B/Cでは人物目標枠と主構図指示を表示すること。
- 指示、撮影ボタン、接続状態が映像を過度に隠さないこと。
- 端末回転、ブラウザUI、セーフエリアが変化しても座標対応を保つこと。

#### 撮影者向け構図指示表示

- 上下左右、近づく・離れる、傾き、構図OKを一貫した位置へ表示すること。
- 主指示は一度に1件とし、補助的な目標枠を併用できること。
- 人物未検出時に方向指示を残さないこと。
- 被写体向けの詳細な関節指示は表示しないこと。
- Ready後も撮影者が任意のタイミングで撮影できることを基本とし、強制ロックの有無は実験設定で統一すること。

#### ガイド表示ON/OFF

- 条件B/Cでのみ操作可能とすること。
- 初期状態を実験プロトコルで固定すること。
- ON/OFF操作をログへ保存すること。
- OFFにしても指示文を残すかどうかを実験設定として固定すること。

#### 撮影・写真確認・送信

- 撮影ボタンは全条件で同じ位置、外観、動作とすること。
- 撮影済み写真のサムネイルを実際に表示すること。
- 個別削除、一括クリア、撮影枚数確認を可能にすること。
- 現時点では送信前確認を挟まず、撮影者が撮影済み写真を全件送信すること。
- 送信成功後に共有写真ID群を受け取り、iOSへ完了通知すること。

#### 条件別表示制御

- 条件制御はサーバー状態を正とし、URLパラメータだけで参加者が変更できないこと。
- 非表示要素はキーボード操作やDOM上の誤操作でも利用できないこと。
- 条件変更・再接続後に古い指示やガイドを残さないこと。
- 条件適用結果を実験者用ログへ記録すること。

| `selectedGuideType` | enum | event / trial | 利用ガイド分析 |

| `endTime` | ISO 8601 | trial | 所要時間 |
| `shotCount` | integer | trial | 撮影効率 |

| `guideTransform` | x, y, scale, opacity | event | ガイド操作分析 |
| `detectedBoundingBox` | x, y, width, height, confidence | inference | 構図差分析 |

### 6.2 推奨イベント共通形式

```json
{
  "schemaVersion": "1.0",
  "eventId": "uuid",
  "participantId": "P012-S",
  "pairId": "PAIR-006",
  "sessionId": "uuid",
  "conditionId": "C",
  "trialId": "uuid",
  "role": "subject",
  "eventType": "instruction_shown",
  "clientTimestamp": "2026-06-18T10:15:30.125+09:00",
  "serverTimestamp": "2026-06-18T01:15:30.180Z",
  "sequenceNumber": 42,
  "payload": {}
}
```

### 6.3 保存構造

最小構成では、既存のセッションディレクトリを次のように拡張する。

```text
uploads/<sessionId>/
  session.json
  trials/<trialId>/
    trial.json
    events.jsonl
    inference.jsonl
    photos/
    guides/
    reference/
    questionnaire/
```

同時書込み、検索、データ件数が増える場合は、メタデータとイベントをSQLiteまたは研究用データベースへ移し、画像だけをファイル保存してもよい。

### 6.4 ログ記録方針

- イベントには単調増加する `sequenceNumber` と一意な `eventId` を付ける。
- 高頻度推定ログと利用者操作ログを分離する。
- 撮影時点の前後にある推定値を特定できるようにする。
- 指示を生成しただけの時刻と、実際に画面表示した時刻を区別する。
- 非表示条件Aでも認識結果を裏側で記録する場合は、同意説明に含める。
- 欠損値を0や空配列で代替せず、`null` と欠損理由を保存する。
- 再送による重複は `eventId` で排除する。
- 実験者による訂正は上書きせず、変更履歴を残す。

### 6.5 研究倫理・プライバシー要件

- 研究開始前に、撮影画像、推定特徴、操作ログ、アンケート、任意の音声記録について個別に同意を得る。
- `participantId` と氏名・連絡先の対応表は研究データと分離し、アクセスを研究責任者等へ限定する。
- 分析用データには氏名、端末名、IPアドレス等を原則として含めない。
- 画像を保存せず特徴量だけで分析可能な項目については、特徴量保存を優先する。
- 顔を含む画像の保存目的、保存期間、閲覧者、発表時の利用方法を同意文書へ明記する。
- 発表資料へ画像を掲載する場合は、実験参加同意とは別に掲載同意を確認する。
- 参加撤回に対応できるよう、参加者単位で関連データを検索・削除できること。
- 保存データは暗号化された研究用ストレージへ移し、端末・開発PC上の一時データを規定期間後に削除する。
- 保存期間は所属機関の倫理審査・研究データ方針に従い、システム設定と研究計画書で具体的な年月を定める。
- 商用運用レベルの認証は対象外だが、実験中は推測困難なセッションID、ローカルネットワーク、アクセス制限されたサーバーを使用する。

## 7. API・WebSocket要件

### 7.1 API追加方針

既存の写真、ガイド、セッションAPIは可能な限り流用する。実験メタデータとログは責務が異なるため、既存APIへ多数の任意項目を追加するのではなく、最小限の実験用エンドポイントを追加する。

### 7.2 推奨HTTP API

| Method | Path案 | 優先度 | 用途 | 既存APIとの関係 |
|---|---|:---:|---|---|
| POST | `/api/experiments/sessions` | P0 | 参加者、ペア、条件順を含む実験セッション作成 | 既存セッションIDを返し、写真APIと共有 |
| GET | `/api/experiments/sessions/:sessionId` | P0 | 再接続時の実験状態取得 | 既存セッション情報の拡張 |
| PATCH | `/api/experiments/sessions/:sessionId/condition` | P0 | 試行前の条件設定 | 試行開始後は拒否 |
| POST | `/api/experiments/sessions/:sessionId/trials` | P0 | `trialId` を発行し試行を構成 | 新規 |
| POST | `/api/experiments/trials/:trialId/start` | P0 | 開始時刻と状態を確定 | 汎用状態更新APIへの統合も可 |
| POST | `/api/experiments/trials/:trialId/end` | P0 | 完了・中断、必要に応じて任意の最終写真IDを記録 | 汎用状態更新APIへの統合も可 |
| POST | `/api/experiments/trials/:trialId/events` | P0 | 操作・指示・Readyログを一括保存 | 複数イベントのバッチ受付を推奨 |
| POST | `/api/experiments/trials/:trialId/inference` | P1 | 高頻度認識結果を保存 | eventsへ統合可能だが分離を推奨 |
| POST | `/api/experiments/questionnaires` | P0 | 条件別・総括アンケート保存 | 新規 |
| PUT | `/api/references/:referenceImageId/pose` | P0 | 参照特徴と手動補正版を保存 | 既存ガイド生成APIの結果を利用 |
| GET | `/api/references/:referenceImageId/pose` | P0 | 採用中の参照姿勢を取得 | 新規 |
| GET | `/api/experiments/export` | P1 | 条件・試行を指定して分析データ出力 | 実験者のみ利用 |

写真アップロード時には、既存 `POST /api/photos` へ `trialId`、`captureTimestamp` を追加する拡張を優先し、重複する写真APIは作らない。現時点では撮影者が撮影済み写真を全件送信するため、`isFinalCandidate` や最終写真選択は必須にしない。

### 7.3 WebSocket通知

既存 `/ws/session` を拡張し、役割をクエリまたは接続直後の登録メッセージで指定する。

| 通知種別案 | 方向 | 内容 |
|---|---|---|
| `experiment-configured` | Backend → 両端末 | 条件、試行、参照写真、支援Level |
| `trial-state-changed` | Backend → 両端末 | 試行状態と時刻 |
| `composition-instruction` | Backend → Web | 撮影者向け構図指示 |
| `pose-instruction` | Backend → iOS | 被写体向けポーズ指示 |
| `readiness-changed` | 双方向 | 役割別Readyと統合Ready |
| `capture-completed` | Web → Backend → iOS | 撮影時刻、写真ID |
| `photos-sent` | Backend → iOS | 送信完了、写真一覧 |
| `inference-status` | Backend → 対象端末 | 検出中、低信頼度、Level低下 |
| `log-sync-status` | Backend → 対象端末 | 受領済みsequence、再送要求 |

### 7.4 通信要件

- 役割別イベントはサーバーで配送先を制御し、クライアント側の非表示だけに依存しない。
- イベントには `sessionId`、`trialId`、`conditionId`、`eventId`、サーバー時刻を含める。
- 古い試行の遅延イベントは現在画面へ反映しない。
- 再接続時は最後に受領したsequence以降を再送するか、最新状態のスナップショットを返す。
- 条件Aでは支援イベントをクライアントへ送らない、または表示不能な明示設定にする。
- ログ送信とリアルタイム支援を分離し、ログ保存遅延によって撮影支援を停止させない。

## 9. 受入条件

システムを「ユーザー実験ができる状態」と判断するには、以下をすべて満たすこと。

### 9.1 実験フロー

- 実機スマートフォン2台とバックエンドを用い、実験セッション作成からアンケート送信まで完了できる。
- QRコードを使って撮影者が正しい `sessionId`、`trialId`、役割へ参加できる。
- 一人の人物が一枚の参照写真を再現する撮影を、条件A/B/Cで実施できる。
- 条件順と参照写真順を実験者が設定できる。
- 練習、本番、中断、再試行を区別できる。

### 9.2 条件操作

- 条件Aでは、撮影者・被写体のどちらにも自動支援が表示されない。
- 条件Bでは、撮影者にのみ構図支援が表示され、被写体にポーズ支援が表示されない。
- 条件Cでは、撮影者に構図支援、被写体にポーズ支援が別々に表示される。
- 条件切り替え後、前条件のガイド、指示、Ready状態が残らない。
- 参加者がURL変更や通常操作で条件を変更できない。

### 9.3 支援機能

- Level 1で人物中心と人物サイズの差分から、上下左右、近づく・離れるを提示できる。
- 少なくともLevel 2で、頭・肩・腰のうち採用した主要点に対する被写体向け指示を提示できる。
- 撮影者向け指示と被写体向け指示が役割の異なる端末へ配送される。
- 低信頼度時に誤指示を継続せず、フォールバックまたは検出不能表示になる。
- 推定機能が停止しても通常撮影へ戻り、写真を撮影・送信できる。

### 9.4 データ完全性

- 撮影結果、共有写真群、認識結果、指示、Ready、アンケートが `sessionId` と `trialId` で紐付く。
- 必須識別子、開始・終了時刻、撮影枚数、撮影時刻、共有写真ID群が完了試行または送信イベントに存在する。
- 条件別に試行時間、撮影枚数、構図誤差、ポーズ誤差、主観評価を比較できる形式で出力できる。
- ログ送信中に一時切断しても、復帰後に必須イベントが重複なく保存される。
- 欠損または中断が発生した場合、その範囲と理由を分析データで識別できる。

### 9.5 UI・運用

- 参加者向け画面にサーバーURL、JSON、信頼度の生値、デバッグログ等が表示されない。
- 撮影済み写真のサムネイルが表示され、個別削除と全件送信ができる。
- 実験者が条件、接続、ログ同期、試行状態を確認できる。
- 失敗時に実験者が試行を中断し、元データを残したまま再試行できる。
- パイロット実験で参加者が画面の次操作を理解でき、重大な操作迷いがない。

### 9.6 倫理・プライバシー

- 同意取得前に本番記録が開始されない。
- 匿名IDと個人情報対応表が分離される。
- 保存期間、削除方法、画像利用範囲が研究計画・同意文書と一致する。
- 参加撤回対象のデータを参加者IDから特定できる。

## 10. 優先度付きロードマップ

### Phase 0: 既存機能の安定化

## 11. 今回は対象外にすること

- 独自AIモデルの学習・ファインチューニング
- 大規模なポーズ推薦モデルの構築
- 参照写真にない新規ポーズの推薦
- 複数人の同時撮影と人物間関係の支援
- 場所、背景、服装に応じたポーズ推薦
- 美しさ、魅力度、写真映え等の美的評価
- 表情、感情、視線の良し悪しの評価
- 顔認識による本人特定
- SNS投稿、公開ギャラリー、共有フィード
- 高度なレタッチ、生成AIによる画像編集
- 音声通話、ビデオ通話
- 発話内容の自動評価
- 商用運用を想定した本格的なアカウント・認証・課金基盤
- クラウド規模の同時多数セッション運用
- Level 3の全関節支援を本実験の必須条件にすること
- LLMが指示内容やReady判定を決定すること

これらは、二者へ役割別支援を提示することの効果を検証する中心課題を不必要に広げ、実験結果の解釈を難しくするため、本研究の基本実験から除外する。
