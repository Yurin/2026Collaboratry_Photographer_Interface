## 2. 背景と目的

一般的な人物撮影では、被写体が「どのように写りたいか」を撮影者へ言葉だけで伝えることが難しい。また、撮影者が参照写真を見ながら構図やポーズを再現する場合でも、カメラ画面上で人物位置や大きさを正確に合わせる作業には負担がある。

本システムは、被写体が選んだ参照写真から撮影用ガイドを作成し、そのガイドを撮影者のカメラ画面へ重ねることで、被写体と撮影者の協調的な撮影を支援することを目的とする。

### 2.1 解決したい課題

- 被写体の希望する構図・人物位置・ポーズを撮影者へ伝えにくい
- 撮影者と被写体が同じ完成イメージを共有しにくい
- 撮影中に被写体から細かな位置調整を伝える負担が大きい
- 撮影後の写真受け渡しに別サービスが必要になる

### 2.2 システムの提供価値

- 参照写真を視覚的なガイドへ変換できる
- 被写体と撮影者が同一セッションでガイドを共有できる
- 被写体が撮影者のライブ映像を確認しながらガイドを調整できる
- 撮影から写真受け渡しまでを一つの流れで行える

## 3. 利用者と利用環境

### 3.1 利用者

| 利用者 | 主な役割 |
|---|---|
| 被写体 | iOS アプリで参照写真選択、ガイド作成、撮影指示、受信写真確認を行う |
| 撮影者 | Web アプリでカメラ撮影、ガイド確認、ライブ共有、写真送信を行う |
| 開発・運用者 | バックエンド起動、接続先設定、保存データ管理を行う |

### 3.2 対象環境

- 被写体側: iOS 18.5 以上を設定した SwiftUI アプリ
- 撮影者側: MediaDevices、Canvas、WebSocket、Fetch API に対応したスマートフォンブラウザ
- サーバー側: Node.js、npm、Python 3
- ガイド生成: Pillow、OpenCV、NumPy、Ultralytics
- 通信環境: iOS 端末と撮影者端末の両方からバックエンドへ到達できること

## 4. システム構成

```text
被写体 iOS アプリ
  ├─ 参照写真・ガイドの端末内保存
  ├─ HTTP API ─────────────┐
  └─ WebSocket（映像受信） ─┤
                            ▼
                    Node.js バックエンド
                      ├─ セッション管理
                      ├─ 写真・ガイド保存
                      ├─ WebSocket 中継
                      ├─ 撮影者 Web 配信
                      └─ Python ガイド生成
                            ▲
  ┌─ HTTP API ─────────────┤
  └─ WebSocket（映像送信・通知受信）
撮影者 Web アプリ
```

## 6. 機能要件

### 6.1 ガイド作成

- FR-G01: 被写体は写真ライブラリから参照写真を選択できること。
- FR-G02: システムは参照写真上に縦3:横4のトリミング範囲を表示すること。
- FR-G03: 被写体はトリミング範囲を移動、拡大、縮小、リセットできること。
- FR-G04: システムはトリミング後の画像から以下のガイドを生成できること。
  - 人物枠
  - 人物枠＋主要ポイント
  - シルエット
- FR-G05: 人物が検出できない場合、処理を異常終了させず代替ガイドを提示すること。
- FR-G06: 被写体は生成した3種類のガイドを切り替えて確認できること。
- FR-G07: 被写体はガイドの左右位置と大きさを調整できること。
- FR-G08: 被写体はタイトルを付けてガイドセットを保存できること。未入力時は既定タイトルを使うこと。

### 6.2 ガイド管理

- FR-L01: 保存済みガイドを新しい順に一覧表示できること。
- FR-L02: 保存済みガイドは参照写真、ガイド画像、タイトル、作成日時を保持すること。
- FR-L03: ガイドの詳細および全画面画像を表示できること。
- FR-L04: ガイドを削除した場合、関連する端末内画像も削除すること。
- FR-L05: 一つ以上のガイドを選択して撮影画面へ渡せること。
- FR-L06: 撮影中に選択済みガイドを切り替えられること。
- FR-L07: 過去形式の単一ガイドデータも可能な範囲で表示できること。

### 6.3 撮影セッション

- FR-S01: 撮影開始ごとに重複しにくいセッション ID を生成すること。
- FR-S02: iOS アプリは撮影者 Web のセッション URL を QR コード表示すること。
- FR-S03: 撮影者 Web は URL の `sessionId` を読み取ること。
- FR-S04: `sessionId` がない場合、撮影機能を継続せず利用者へ案内を表示すること。
- FR-S05: iOS アプリ内の受信写真画面は現在のセッション ID を参照すること。
- FR-S06: セッション終了後も、削除または期限切れまでは送信済み写真を取得できること。

### 6.4 ライブ映像

- FR-V01: 撮影者 Web は利用者の許可を得てカメラを起動すること。
- FR-V02: ライブ共有の開始・停止を撮影者が操作できること。
- FR-V03: Web は映像を圧縮した JPEG フレームとして一定間隔で送信すること。
- FR-V04: バックエンドは同一セッションの送信者から受信者へフレームを中継すること。
- FR-V05: iOS アプリは最新受信フレームを画面表示すること。
- FR-V06: 切断時には接続状態を表示し、再接続を試みること。

### 6.5 ガイド共有・撮影支援

- FR-O01: iOS アプリは撮影者へ送るガイド種類を選択できること。
- FR-O02: iOS アプリ内で見るガイドと、撮影者へ送るガイドを別々に選択できること。
- FR-O03: ガイド画像を更新した場合、Web にリアルタイム通知すること。
- FR-O04: Web はカメラ映像上に透過 PNG ガイドを重ねること。
- FR-O05: 被写体は撮影者画面上のガイドについて、透明度、左右位置、大きさを調整できること。
- FR-O06: Web は受信した調整値を即時反映し、変更通知を表示すること。
- FR-O07: 撮影者はガイド表示の ON/OFF と透明度をローカル変更できること。
- FR-O08: すべてのガイド表示領域は原則として3:4の撮影領域を基準とすること。

### 6.6 写真撮影・受け渡し

- FR-P01: 撮影者は現在のカメラ映像から静止画を撮影できること。
- FR-P02: 撮影者は未送信写真の枚数を確認できること。
- FR-P03: 撮影者は未送信写真を一覧表示し、個別削除できること。
- FR-P04: 撮影者は未送信写真を一括クリアできること。
- FR-P05: 撮影者は複数写真を一括送信できること。
- FR-P06: バックエンドは写真をセッション単位で保存すること。
- FR-P07: iOS アプリは現在のセッションの写真のみを取得すること。
- FR-P08: 被写体は受信写真を1枚または一括で写真ライブラリへ保存できること。
- FR-P09: 保存権限がない場合、理由を利用者へ表示すること。

### 6.7 データ削除

- FR-D01: 撮影者は確認操作後、現在のセッションのサーバー保存データを削除できること。
- FR-D02: バックエンドは設定時間を超えたセッションデータを自動削除すること。
- FR-D03: セッション削除時、接続中の Web へ削除を通知すること。
- FR-D04: 削除後の Web 画面はガイドと未送信写真表示を初期化すること。

## 7. データ要件

### 7.1 iOS 端末内データ

保存先はアプリの Documents ディレクトリとする。

| データ | 保存形式 |
|---|---|
| ガイド一覧メタデータ | `guides.json` |
| 参照写真 | `reference_<UUID>.jpg` |
| 人物枠ガイド | `guide_rectangle_<UUID>.png` |
| 主要ポイントガイド | `guide_keypoints_<UUID>.png` |
| シルエットガイド | `guide_silhouette_<UUID>.png` |

ガイドメタデータは、ID、タイトル、各画像の相対パス、作成日時を保持する。

### 7.2 サーバー保存データ

- 保存先: `backend/uploads/<安全化したsessionId>/`
- 種別ごとの保存先:
  - 撮影写真: `photos/`
  - 参照写真: `references/`
  - 生成・共有ガイド: `guides/`
- セッション ID に使えない文字は `_` に置換する
- 既定の保持時間: 6時間
- 既定の削除確認間隔: 10分

### 7.3 データ上の改善要件

新規データはサブディレクトリで種別を分離する。旧形式のセッション直下ファイルについては、ファイル名による判別で後方互換性を保つ。実験ログを追加する段階では、セッションメタデータ JSON またはデータベースによる管理へ拡張する。

## 8. API・通信要件

### 8.1 HTTP API

| Method | Path | 用途 |
|---|---|---|
| GET | `/` | 撮影者 Web を返す |
| GET | `/health` | サーバー起動確認 |
| POST | `/api/photos` | セッション ID と写真群を保存 |
| GET | `/api/photos?sessionId=...` | セッションの写真一覧を取得 |
| POST | `/api/session/:sessionId/guide` | 共有ガイドを保存 |
| GET | `/api/session/:sessionId/guide` | 最新ガイドを取得 |
| POST | `/api/session/:sessionId/generate-guide` | 指定した1種類のガイドを生成 |
| POST | `/api/session/:sessionId/generate-guide-set` | 3種類のガイドを一括生成 |
| DELETE | `/api/session/:sessionId` | セッション保存データを削除 |

### 8.2 WebSocket

| Path | 主な方向 | 用途 |
|---|---|---|
| `/ws/video?sessionId=...&role=sender` | Web → backend | ライブ映像送信 |
| `/ws/video?sessionId=...&role=receiver` | backend → iOS | ライブ映像受信、iOSからのガイド調整送信 |
| `/ws/session?sessionId=...` | backend → Web | 接続、ガイド更新、ガイド調整、削除通知 |

### 8.3 主な通知種別

- `connected`
- `guide-updated`
- `guide-transform`
- `photos-updated`
- `session-deleted`

## 9. 非機能要件

### 9.1 操作性

- NFR-U01: 初めての利用者が「ガイド作成→QR共有→撮影→写真保存」の順序を理解できる表示にすること。
- NFR-U02: 処理中はボタンの無効化または進捗表示を行い、二重操作を防ぐこと。
- NFR-U03: エラーは開発者向け情報ではなく、利用者が次に行う操作を日本語で示すこと。
- NFR-U04: 主要操作はスマートフォンの片手操作を考慮したサイズと配置にすること。

### 9.2 性能

- NFR-P01: ライブ映像は撮影指示に利用できる遅延に抑えること。目標値は通常環境で1秒以内とする。
- NFR-P02: ガイド調整は送信後500ms以内に撮影者画面へ反映されることを目標とする。
- NFR-P03: ガイド生成中は進捗状態を表示し、タイムアウト時は再試行可能にすること。
- NFR-P04: 高解像度写真の一括送信でブラウザが停止しないよう、枚数・画像サイズを制御できること。

### 9.3 信頼性

- NFR-R01: WebSocket 切断時に再接続を試みること。
- NFR-R02: 一つの不正画像やガイド生成失敗によってサーバープロセス全体が停止しないこと。
- NFR-R03: 保存・削除 API は結果を明示すること。
- NFR-R04: 端末内ガイド情報と画像ファイルの不整合を可能な範囲で回復または通知すること。

### 9.4 セキュリティ・プライバシー

- NFR-S01: 公開・学外利用時は HTTPS / WSS を使用すること。
- NFR-S02: セッションへ推測困難な ID を使用すること。
- NFR-S03: 本運用ではセッション PIN、期限付きトークン等のアクセス制御を追加すること。
- NFR-S04: アップロード可能な MIME type、ファイルサイズ、枚数を検証すること。
- NFR-S05: CORS は利用するオリジンに限定すること。
- NFR-S06: 保存目的と削除時期を利用者へ説明すること。
- NFR-S07: ログへ画像内容や不要な個人情報を出力しないこと。

### 9.5 保守性・可搬性

- NFR-M01: サーバー URL、WebSocket URL、ポート、保存期限を設定値として変更できること。
- NFR-M02: 固定 LAN IP に依存せず、環境別設定を切り替えられること。
- NFR-M03: API、ガイド生成、主要 UI フローに自動テストを追加できる構造にすること。
- NFR-M04: 生成モデル名と閾値をコード変更なしで設定できることが望ましい。

## 10. 現在の制約・既知課題

| 優先度 | 課題 | 影響 |
|---|---|---|
| 高 | ポーズ推定結果が参照写真によって安定しない | 希望する手足の姿勢を正確にガイド化できない |
| 高 | 認証・認可がない | セッション URL を知る第三者がデータへアクセスできる |
| 中 | 実機では設定アプリへサーバーURLを入力する必要がある | 入力漏れや誤入力時に接続できない |
| 中 | HTTP / WebSocket が平文通信 | 公開環境ではカメラ利用制約と盗聴リスクがある |
| 中 | アップロードサイズ、形式、枚数の上限がない | メモリ・ディスク消費や不正ファイルのリスクがある |
| 中 | ガイドセット生成時に Python 処理を3回起動する | モデル読込を含む場合、生成時間が長くなりやすい |
| 中 | 最新ガイドをファイル名の文字列順で決定する | 同一ミリ秒や命名変更時に誤選択する可能性がある |
| 中 | ライブ映像送信用 WebSocket は自動再開しない | 切断後、撮影者の再操作が必要 |
| 低 | API エラー文が英語・固定文中心 | 原因の切り分けが難しい |
| 低 | テストが実質的に未整備 | 修正時の回帰不具合を検出しにくい |

## 11. 今後できそうなこと

### 11.1 優先度A: 現行機能を研究・デモで安定させる

1. 撮影済みサムネイルを画面内に表示し、個別削除を利用可能にする。
2. サーバー保存先を写真・参照画像・ガイドに分離する。
3. LAN 内の実機2台を使い、ガイド作成から写真保存までの結合試験を作る。
4. ガイド生成失敗、WebSocket 切断、権限拒否の再試行導線を整える。
5. 接続先 URL を開発・実験環境ごとに切り替えられるようにする。
6. 写真枚数、画像サイズ、MIME type の制限を追加する。

### 11.2 優先度A: 手描き棒人間・ポーズ補正

自動ポーズ推定が不正確な場合に備え、利用者がガイドを直接補正できる機能を追加する。

想定フロー:

1. 参照写真と自動推定された関節点を表示する。
2. 利用者が頭、肩、肘、手首、腰、膝、足首をドラッグして修正する。
3. 検出できない部位はタップで追加する。
4. 関節点を線で結び、棒人間としてプレビューする。
5. 棒人間、人物枠、シルエットを別レイヤーとして ON/OFF する。
6. 修正後の座標データと透過 PNG を保存し、撮影者へ共有する。

最初の実装では完全なお絵描き機能より、固定された関節点を動かす方式が適している。自由描画よりデータを再編集しやすく、評価実験でも「どの部位をどれだけ修正したか」を記録できるためである。

追加データ例:

```json
{
  "canvasAspectRatio": "3:4",
  "joints": {
    "leftShoulder": {"x": 0.42, "y": 0.28},
    "leftElbow": {"x": 0.31, "y": 0.42}
  },
  "source": "auto-corrected"
}
```

### 11.3 優先度B: 撮影支援の高度化

- 被写体から撮影者へ定型メッセージを送る
  - 「もう少し右」
  - 「近づいて」
  - 「カメラを下げて」
- ガイド中心と人物位置の差を検出し、方向矢印で提示する
- 一致度をスコアまたは色で表示する
- 水平線、三分割線、顔位置、余白ガイドを選べるようにする
- 複数人の人物ガイドへ対応する
- 前面・背面カメラ、ズーム、ライトを Web から操作する
- 撮影時にガイド有無の比較画像を保存する

### 11.4 優先度B: ガイド生成改善

- 自動検出された関節点を「主要ポイント」ガイドにも使用する
- 複数モデル・閾値を比較し、写真条件別の失敗率を計測する
- 1回の Python プロセスで3種類のガイドを生成する
- モデルを常駐させ、連続生成時の待ち時間を短縮する
- 人物未検出時に範囲指定や手動作成へ誘導する
- 背景、遮蔽、座位、全身・上半身などの条件別評価を行う

### 11.5 優先度C: 運用・共有機能

- セッション PIN またはワンタイムトークン
- 撮影者の参加・退出表示
- 複数撮影者への対応
- セッション履歴、撮影枚数、所要時間の記録
- お気に入り、タグ、検索、ガイドの並び替え
- ガイドや撮影結果のクラウド同期
- 写真ごとの選択、評価、コメント
- 保存期限の表示と利用者による延長

## 13. 受入条件

### 13.1 最小実用版

以下を一連の操作として実機で完了できること。

1. 被写体が参照写真を選択する。
2. 3:4にトリミングしてガイドを生成する。
3. ガイドを保存し、撮影用として選択する。
4. 撮影者が QR コードから Web へ参加する。
5. 撮影者のライブ映像が iOS に表示される。
6. 被写体が共有したガイドが撮影者のカメラ上に表示される。
7. 被写体側の位置・大きさ・透明度変更が撮影者側へ反映される。
8. 撮影者が複数枚を撮影し、一覧確認後に送信する。
9. 被写体が受信写真のみを一覧表示し、写真ライブラリへ保存する。
10. セッションデータを手動または期限切れで削除できる。

### 13.2 ポーズ補正版

- 自動推定された主要関節点を参照写真上に表示できること。
- 利用者が関節点を移動・追加・削除できること。
- 修正結果から棒人間ガイドを再生成できること。
- 修正したガイドを保存後に再編集できること。
- 修正ガイドを撮影者 Web に共有できること。

## 14. 対象外または将来判断とする事項

- 音声・ビデオ通話
- SNS 投稿
- 写真の高度なレタッチ
- 決済
- 不特定多数向けの公開ギャラリー
- AI による美的評価や容姿評価

これらは本研究の中心である「参照写真を用いた被写体と撮影者の協調支援」への寄与を確認した後に、必要性を判断する。
